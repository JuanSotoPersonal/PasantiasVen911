1. Visión General

El sistema VEN 911 Carabobo es una plataforma de software privativa diseñada para administrar el ciclo de vida de fichas de emergencias, despachos informativos y gestión de usuarios departamentales.

Ha sido desarrollada descartando ecosistemas hiper-abstractos (como Laravel o Symfony) en favor de un patrón MVC (Modelo-Vista-Controlador) puro, ágil y de Inercia Cero. El sistema responde estrictamente a las solicitudes delegando la lógica de frontales a scripts centralizados Javascript purgados y la responsabilidad de los datos a un backend impenetrable en PHP apoyado por MariaDB.

💡

Stack Tecnológico

Backend: PHP 8.2 (Vanilla OOP + PDO + MVC)

Base de Datos: MariaDB 10.4

Frontend: HTML5, CSS3, JS ES6+, AdminLTE 4, Bootstrap 5

Plugins JS: DataTables, SweetAlert2, ApexCharts

2. Arquitectura MVC

2.1. Router Frontal (Front Controller)

Toda la aplicación es controlada mediante el archivo index.php de la raíz del servidor. Bloquea todo intento de navegación paralela.

// Ejemplo de acceso URL:
http://localhost/ProyectoFicha/index.php?url=usuario/guardar

[REQUEST GET/POST]
       |
1. index.php intercepta parámetro "url".
2. Corta por separador "/": Controlador = "Usuario", Método = "Guardar".
3. Identifica si hay ID (Ej: usuario/editar/4).
4. RBAC: ¿Existe $_SESSION['user_id']? ¿Tiene permiso de VER este archivo? 
5. Si pasa -> Carga el archivo respectivo instanciando la clase e invoca dinámicamente.

2.2. Estructura de Directorios

Cumple con el patrón Cero Fricciones; un idioma común (Español) y abstracciones justas:

  • /app/controladores/: Gestionan la recolección $_POST y dictaminan el flujo. Tienen Sello Hermético en constructores frente a infiltraciones.
  • /app/modelos/: Repositorios PDO exclusivos. Nunca asumen lógica de negocio, se limitan a SELECT, INSERT, UPDATE, siempre devolviendo DataArrays.
  • /app/vista/: Código nativo HTML. Cargan partials/ (sidebar, footer) y no contienen validaciones lógicas directas.
  • /app/Helpers/: Contiene librerías estáticas de abstracción como `Validador.php`.

4. Seguridad de Fronteras

4.1. Regla "Inercia Cero"

Las anteriores versiones del sistema delegaban controles lógicos al cliente a través de atributos perezosos nativos del DOM (como minlenght="6" o required en los inputs). Esto genera falsas confianzas al programador.

🛡️

Frontend Despojado

Todas las restricciones HTML nativas fueron abolidas de las vistas. Es el Backend el único amo y señor de los datos recibidos. Si el usuario modifica el DOM o el JavaScript por consola, o desactiva las restricciones del explorador para someter datos en crudo, el Backend filtrará, desechará y expulsará estas peticiones silenciando el fallo elegante en JSON.

4.2. Helper de Validación (App\Helpers\Validador)

Las comprobaciones ahora están agrupadas paramétricamente dentro del Validador con una lógica ineludible e irrevocable.

// Dentro del Controlador
$validacionClave = Validador::validarContrasena($_POST['pass']);

if (!$validacionClave['valido']) {
    echo json_encode(['success' => false, 'message' => $validacionClave['mensaje']]);
    return;
}
// Prosigue sin miedos y esterilizado.

5. Autenticación y RBAC

Se sustituyó la noción abstracta de IDs flotantes permitiendo escudos inquebrantables. El Control de Accesos Basado en Roles (RBAC) tiene una jerarquía predeterminada:

  1. Administrador: Domina sin fricciones.
  2. Jefatura: Visualización Total, Modificación Denegada. Módulo "Reportes" asignado nativamente.
  3. Operador: Recepción y Creación. No despacha, ni visualiza historiales masivos.
  4. Despachador: Recoge el objeto Ficha, Modifica su bitácora viva, lo asigna a entes policiales.

Todo controlador invoca un __construct() obligatorio:

public function __construct() {
    if (session_status() === PHP_SESSION_NONE) {
        session_start();
    }
    // Denegación por Defecto. Silencio Absoluto. Cero Piedad.
    if (!isset($_SESSION['user_id'])) {
        header('Location: index.php?url=auth');
        exit;
    }
}

6. Sistema de Auditoría Multidimensional

¿Qué pasa cuando la Ficha ID #43 es modificada y eliminada por otra persona? Las tablas tradicionales apilan toda la chatarra en una misma base lo que degrada el perfomance de las uniones y rastreos.

Para esto nacieron dos mundos:

  • eventos_fichas: Responde a la pregunta ¿Qué vida tuvo este documento? Aquí se registran cambios de Estado (Ingresó -> Despachado). Requiere ficha_id mandatorio.
  • eventos_sistema: Responde a ¿A qué horas entra un usuario al sistema? ¿Qué configuraciones generales altera? Operaciones desvinculadas de la emergencia viva.

8. Arquitectura de Peticiones DataTables Server-Side (Obligatoria)

Por políticas directas de rendimiento frente a grandes volúmenes de data (como Bitácoras policiales de 10.000 Fichas Diarias), está estrictamente prohibido y penado construir tablas realizando la petición de una carga nativa SQL completa en bloque.

Toda DataTables instanciada en window.Ven911DataTablesLang utiliza Backend dinámico. He aquí su anatomía:

8.1. Parámetros del Formulario Dinámico (Payload Oculto)

Cuando el JavaScript de Datatables se arranca, no pide una página HTML, sino que genera variables `POST` automáticamente bajo demanda:

  • draw: Número entero ascendente. Activa candado anti-Asicronía (asegurar el orden de render).
  • start: Puntero Base (Equivalente al OFFSET SQL).
  • length: Límite a extraer (Equivalente al LIMIT SQL).
  • order[0][column]: Entero que el programador asocia a una columna real.
  • order[0][dir]: Criterio DESC / ASC.
  • search[value]: Cadena introducida en el instante de tecleo de la barra universal.

8.2. Filtrado Inteligente en Contexto Backend (Controlador y Modelo)

El código central del sistema agrupa estas órdenes en la capa XControlador.php y acude al modelo bajo un hilo secuencial de 3 Fases Lógicas:

  1. Conteo Bruto: Genera un COUNT(*) de la Fila Madre que le permite conocer al sistema la profundidad total de registros que tendría que arrojar teóricamente. (Rápido y limpio).
  2. Conteo Mermado (Condicional Filtrada): Un segundo COUNT(*) que verifica si el índice search[value] dictó algo. Si es el caso, determina el tamaño actual de las páginas virtuales usando un LIKE "%$busqueda%".
  3. Cosecha del Nodo Finito: El PDO Final y agresivo. Acuarela estrictamente la columna (`whitelist`), impone limitadores `LIMIT` y `OFFSET` calculados al pelo y trae, de 10.000 registros, únicamente y milimétricamente las 10 míseras filas solicitadas. Todo sin sobrecargar la RAM del Servidor Apache.
