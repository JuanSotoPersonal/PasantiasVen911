2.2 Flujo de Enrutamiento (Front Controller)

Toda petición HTTP llega a index.php, que actúa como controlador frontal aplicando autenticación y enrutamiento:

GET index.php?url=usuario/guardar
         │
         ▼
   1. Sanitizar URL (FILTER_SANITIZE_URL)
   2. Verificar sesión ($_SESSION['user_id'])
      ├── No autenticado + ruta protegida → redirect auth
      └── Autenticado + ruta auth → redirect home
         │
         ▼
   3. Resolver controlador: "usuario" → UsuarioControlador
   4. Resolver método: "guardar"
   5. call_user_func_array([$controller, $method], $params)
💡
Futuro: Se recomienda integrar en este flujo una verificación de permisos RBAC que consulte la tabla rol_permiso desde la sesión, bloqueando accesos no autorizados a nivel de controlador.

3.2 Correcciones Aplicadas en Detalle

🔧 XSS en DataTables — escapeHTML()

Se introdujo una función de escape universal en ambos archivos JS que convierte los caracteres HTML peligrosos a entidades seguras antes de inyectarlos al DOM:

const escapeHTML = (str) => {
  if (typeof str !== 'string' && str != null) str = str.toString();
  if (!str) return str;
  return str.replace(/[<>'"&]/g, tag => ({
    '&': '&amp;', '<': '&lt;', '>': '&gt;',
    "'": '&#39;',     '"': '&quot;'
  }[tag] || tag));
};

Adicionalmente, en logs_datatable.js se reemplazó el uso inseguro de .html(formatted) por la creación dinámica con $('<pre>').find('code').text(formatted), que escapa todo contenido JSON automáticamente.

🔧 Fuga de Error en Base de Datos — Database.php

// ANTES (exponía detalles internos del servidor al navegador)
die("Error de conexión a la base de datos: " . $exception->getMessage());

// DESPUÉS (mensaje genérico al usuario, detalle al log del servidor)
error_log("Database connection error: " . $exception->getMessage());
die("Error de conexión a la base de datos.");

🔧 Política de Contraseñas — Mínimo 8 Caracteres

Se actualizaron las validaciones de longitud en 3 controladores:

  • AuthControlador.php → método authenticate()
  • RegistroControlador.php → método registrar()
  • UsuarioControlador.php → métodos guardar() y actualizarContrasena()

Los requisitos actuales de contraseña son: mínimo 8 caracteres, máximo 128, al menos una mayúscula y un número.

Sección 4

Control de Acceso Basado en Roles (RBAC)

El sistema anterior usaba un único campo rol_id en la tabla usuarios para inferir el nivel de acceso. Este approach es rígido: para cambiar qué puede hacer un rol es necesario modificar el código PHP. El nuevo modelo RBAC delega esa lógica a la base de datos.

4.1 Nuevas Tablas — Esquema Relacional

roles ─────────┐
               │  1:N
           rol_permiso (pivote)
               │  N:1
permisos ──────┤
               │  N:1
modulos ───────┘

modulos: Categorías funcionales del sistema (ej. fichas, usuarios).

permisos: Acciones específicas dentro de un módulo (ej. fichas.crear, fichas.editar).

rol_permiso: Tabla pivote que vincula qué permisos tiene cada rol, usando claves primarias compuestas para evitar duplicados.

Sección 6

Eventos: Campos id_user e id_owner

La tabla fichas_emergencia es la entidad central del sistema — representa cada emergencia gestionada. Para implementar trazabilidad completa y un modelo de propiedad de evento, se añaden dos nuevas columnas:

👤 id_user — Creador

  • FK a usuarios.id
  • Se asigna al momento de crear la ficha
  • Nunca cambia — registro inmutable del origen
  • ON DELETE SET NULL

🔄 id_owner — Propietario actual

  • FK a usuarios.id
  • Se actualiza cada vez que se modifica la ficha
  • Indica quién es el "responsable activo" del evento
  • ON DELETE SET NULL

Ciclo de vida de una ficha

Operador crea ficha          → id_user = Operador.id, id_owner = Operador.id
Despachador asigna unidad    → id_owner = Despachador.id
Jefatura añade observaciones → id_owner = Jefatura.id
Ficha cerrada                → id_owner conserva el último editor
⚠️
Al actualizar la ficha, el modelo PHP debe asignar id_owner = $_SESSION['user_id'] automáticamente, y si el campo id_user es nulo (fichas migradas), asignarlo en la primera edición post-migración.

SQL de migración

ALTER TABLE `fichas_emergencia`
  ADD COLUMN IF NOT EXISTS `id_user`  INT UNSIGNED DEFAULT NULL
    COMMENT 'Usuario que CREA la ficha (inmutable)'
    AFTER `caso_id`,
  ADD COLUMN IF NOT EXISTS `id_owner` INT UNSIGNED DEFAULT NULL
    COMMENT 'Ultimo usuario que MODIFICO la ficha'
    AFTER `id_user`;

ALTER TABLE `fichas_emergencia`
  ADD CONSTRAINT `fk_ficha_id_user`  FOREIGN KEY (`id_user`)
    REFERENCES `usuarios` (`id`) ON DELETE SET NULL,
  ADD CONSTRAINT `fk_ficha_id_owner` FOREIGN KEY (`id_owner`)
    REFERENCES `usuarios` (`id`) ON DELETE SET NULL;
✅
v3.0 — Simplificación: Tras la auditoría de redundancia, operador_id fue eliminado de fichas_emergencia (reemplazado por id_user) y despachador_id fue eliminado (reemplazado por id_owner). Ver §9 para el detalle completo.

Cambios v3.1 — RBAC, SSE Fix & Refactorización

Esta sección documenta todos los cambios aplicados durante la sesión del 13 de Abril de 2026 que elevan el sistema a la versión 3.1. Los cambios abarcan correcciones críticas de bugs, implementación del middleware RBAC completo y refactorización integral del módulo de usuarios.

10.1 Fix Crítico: Session Lock en SSE

🚨
Bug reportado: Al navegar al módulo de Usuarios, la página quedaba cargando indefinidamente. El sistema estaba funcionalmente bloqueado.

Causa raíz

PHP utiliza archivos para el almacenamiento de sesiones. Mientras un script PHP mantiene la sesión abierta con session_start(), bloquea el archivo de sesión para escritura exclusiva. El loop SSE de NotificacionControlador::stream() entra en un while(true) con sleep(6) y nunca libera ese lock, causando que cualquier otra pestaña o petición que llame a session_start() quede bloqueada esperando.

Corrección aplicada

// NotificacionControlador.php - método stream()
$usuario_id = (int)$_SESSION['user_id'];
session_write_close(); // ← Libera el lock ANTES del bucle SSE

set_time_limit(0);
while (true) {
    if (connection_aborted()) break;
    $notificaciones = $this->modelo->obtenerNoLeidas($usuario_id);
    echo "data: " . json_encode($notificaciones) . "\n\n";
    if (ob_get_level() > 0) ob_flush();
    flush();
    sleep(6);
}
✅
session_write_close() libera el lock inmediatamente después de leer user_id. El SSE continúa funcionando porque ya no necesita la sesión activa para operar.

10.2 RBAC Middleware Implementado

Se implementó el sistema completo de Control de Acceso Basado en Roles (RBAC) con carga temprana de permisos en sesión. El patrón elegido es de carga única al login: una sola query a la BD al autenticarse, los permisos se almacenan en $_SESSION['permisos'] y se consumen en todas las peticiones siguientes sin overhead adicional.

Flujo completo

[Login exitoso]
  → AuthControlador::authenticate()
  → $_SESSION['permisos'] = obtenerPermisosDeRol(rol_id)
  → Resultado: ['fichas' => ['ver','crear'], 'usuarios' => ['ver','gestionar'], ...]

[Cada petición]
  → index.php middleware
  → tienePerm('modulo', 'permiso') // consulta solo $_SESSION, sin BD
  → true / false
  → Acceso permitido / redirect a home

Helper global tienePerm()

// index.php
function tienePerm(string $modulo, string $permiso = 'ver'): bool {
    return isset($_SESSION['permisos'][$modulo])
        && in_array($permiso, $_SESSION['permisos'][$modulo], true);
}

Mapa de rutas protegidas (RBAC Gate)

// index.php — RBAC Middleware
$rutasProtegidas = [
    'usuario'      => ['usuarios', 'ver'],
    'log'          => ['historial', 'ver'],
    'notificacion' => ['fichas',   'ver'],
];
$claveRuta = strtolower($controllerBaseName);
if (isset($rutasProtegidas[$claveRuta])) {
    [$moduloReq, $permisoReq] = $rutasProtegidas[$claveRuta];
    if (!tienePerm($moduloReq, $permisoReq)) {
        header('Location: index.php?url=home');
        exit;
    }
}

              JOIN modulos  m ON p.modulo_id   = m.id
              WHERE rp.rol_id = :rol_id";

}

Sección 11 — Nueva en v3.2

Paginación Server-Side — Módulo de Logs

El módulo de historial ahora procesa la paginación, búsqueda y ordenamiento en el servidor (MariaDB), en lugar del navegador del cliente. Esto garantiza tiempos de respuesta constantes sin importar el volumen de registros de auditoría acumulados.

📦 Modo anterior (Client-Side)

  • El servidor enviaba todos los registros en un solo JSON
  • DataTables filtraba y paginaba en el navegador con JavaScript
  • Con 10.000 registros → carga lenta + alto consumo de RAM
  • La búsqueda era instantánea una vez cargado, pero la carga inicial era el cuello de botella

⚡ Modo nuevo (Server-Side)

  • DataTables envía start, length, search y order en cada petición POST
  • El servidor devuelve solo la página solicitada
  • Con 10.000 registros → carga siempre de ~10 filas en <50ms
  • El filtrado y ordenamiento usan índices de MariaDB, no RAM del cliente

11.1 Motivación y Arquitectura

Los logs de auditoría crecen indefinidamente con el uso normal del sistema. Una sola sesión activa de un año puede generar decenas de miles de entradas. La migración a server-side es una medida preventiva de escalabilidad.

Flujo de datos (server-side)

[Navegador] → POST index.php?url=log/obtenerDatos
             → { draw:1, start:0, length:10, search:{value:''}, order:[{column:4,dir:'desc'}] }
                  ↓
[LogControlador::obtenerDatos()]
  → Lee parámetros POST
  → Mapea índice de columna a nombre SQL
  → Llama LogModelo::obtenerPaginado()
  → Llama LogModelo::contarTodos()
  → Llama LogModelo::contarFiltrados() (solo si hay búsqueda)
                  ↓
[MariaDB]
  SELECT ... LIMIT 10 OFFSET 0
  ORDER BY l.fecha DESC
  WHERE accion LIKE '%texto%' OR ...
                  ↓
← JSON: { draw, recordsTotal, recordsFiltered, data:[10 filas] }

11.2 Cambios en LogModelo.php

Se eliminó el método obtenerTodos() (que retornaba todos los registros sin límite) y se reemplazó por tres métodos especializados:

MétodoFirmaDescripción
obtenerPaginado() (int $inicio, int $cantidad, string $busqueda, int $colOrden, string $dirOrden): array Devuelve solo los registros de la página actual. Incluye LIMIT, OFFSET, búsqueda multi-campo y ordenamiento dinámico.
contarTodos() (): int Retorna el total de registros sin ningún filtro. Necesario para recordsTotal.
contarFiltrados() (string $busqueda): int Retorna el total de registros que coinciden con la búsqueda. Necesario para recordsFiltered.

Whitelist de columnas (seguridad anti-SQL injection)

El ordenamiento dinámico podría ser un vector de SQL injection si se inserta el nombre de columna directamente. Se usa un mapa cerrado para prevenir esto:

$columnas = [
    0 => 'l.accion',
    1 => 'l.tabla_afectada',
    2 => 'l.registro_id',
    3 => 'u.usuario',
    4 => 'l.fecha',
];
$columnaOrden = $columnas[$colOrden] ?? 'l.fecha'; // fallback seguro

Búsqueda multi-campo

WHERE (:busqueda = ''
    OR l.accion         LIKE :b1
    OR l.tabla_afectada LIKE :b2
    OR u.usuario        LIKE :b3
    OR l.detalles       LIKE :b4
)

La condición :busqueda = '' permite que el optimizador de MariaDB evalúe en cortocircuito y omita los LIKE cuando no hay búsqueda, evitando un full table scan innecesario.
